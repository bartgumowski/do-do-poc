# Do-Do Code Audit - 2026-06-10

Scope: full repo at v0.7.0. Architecture, security, strategy, optimisation. Read-only review - no code was changed. Every finding below was verified directly against the source.

---

## Executive summary

The product is impressively complete for a no-build vanilla JS POC: clean RLS design, GDPR endpoints, Stripe, push, AI, i18n. The frontend data isolation (Supabase RLS) is the strongest part of the system.

The weakest part is the **API layer: most serverless endpoints have no authentication**. The RLS work is effectively bypassed by service-role endpoints that trust client-supplied IDs. Before any real users beyond the pilot pair, fix the CRITICAL and HIGH items. Estimated effort: 1-2 days, mostly adding one shared `requireUser(req)` helper to existing endpoints.

## 1. Security

### CRITICAL

**S1. `/api/stripe-portal` - IDOR, full billing portal takeover.**
Accepts `{ userId }` from the request body with no auth check, then creates a Stripe Customer Portal session for that user and returns the URL. Anyone who knows or guesses a userId (UUIDs leak via realtime payloads, exports, support screenshots) gets the victim's invoices, payment methods, and can cancel their subscription.
Fix: derive userId from the Supabase JWT (same pattern already used in `delete-account.js`), never from the body.

**S2. `/api/stripe-checkout` - unauthenticated payment-request forgery.**
No auth on either action. `action: "expense"` accepts arbitrary `cardId`, `amount`, `requestedByName`, creates a real PaymentIntent, overwrites the card's `payment_status`/`payment_amount` via service role, and **sends a payment email to the co-parent** with attacker-controlled name and amount. This is a ready-made phishing/fraud channel using your own domain and Resend account. `action: "create"` similarly accepts arbitrary userId/pairId.
Fix: require JWT; verify the authenticated user belongs to the card's pair before creating the intent.

**S3. Stored XSS between partners via messages.**
`features.js` `renderRealMessage()` interpolates `${msg.body}` into innerHTML unescaped. Message bodies are user input (up to 2000 chars per the DB check) synced in realtime to the co-parent. One partner can execute JS in the other's session - in a co-parenting app that is an account-takeover and surveillance vector between the two people the product promises to keep safe. Same gap: `${card.details}` and `${card.title}` in the expense preview card (features.js ~1359-1360) and the fallback `renderUniversalFeatureCard` (~1973-1975). The main `renderUnifiedCard` in app.js escapes correctly, so the helpers exist - they just aren't used everywhere.
Fix: route every user-originated string through `escapeHtml()`; add a CSP header as a backstop (see S8).

### HIGH

**S4. `/api/push-subscribe` - notification hijack.**
No auth. POST accepts any `user_id` + attacker's own push endpoint, upserted via service role. The attacker then receives the victim's reminder notifications (titles include card titles - schedules, medical info). DELETE allows removing any subscription by endpoint. Fix: require JWT, take user_id from the token.

**S5. `/api/ai` - open, unauthenticated Anthropic proxy.**
CORS `*`, no auth, no rate limit. Anyone can burn your API credits from any website, and the `interpret` prompt happily processes arbitrary text. Fix: require JWT + basic per-user rate limit.
